## 1. Slot Fetching — `getAvailableSlots(daysWindow, searchFromDate)`

- Query Google Calendar **freebusy API** for the given window
- Generate Mon–Fri slots, 9am–5pm **Eastern Time** (auto-detect EST vs EDT based on DST)
- DST rule: EDT (UTC-4) = 2nd Sunday March through 1st Sunday November; EST (UTC-5) = rest of year
- Skip slots that are in the past
- Return **up to 6 slots**, each with:
  ```js
  { label: "Tuesday, Mar 10 at 10:00 AM EDT", start: "<ISO UTC string>", end: "<ISO UTC string>" }
  ```
- Label format: `"Weekday, Mon D at H:MM AM/PM EDT"` — **exact, never relative**
- When `searchFromDate` is provided (user asked for specific date): set `daysWindow = 1` and start FROM that exact date (not the day before)
- When no `searchFromDate`: fetch next 7 days from today

---

## 2. Conversation Slot Cache — `conversationSlots` Map

- Key: `conversationId` (UUID generated by the frontend, passed in every request)
- Value: `{ slots: [...], fetchedAt: Date.now() }`
- **Cache expiry: 30 minutes**
- **Cache-first logic:**
  - If cache exists AND covers the date the user is asking about → reuse it (never re-fetch on confirmation messages)
  - If cache is empty OR doesn't cover the requested date → fetch fresh for that date
- **No slot reservation system** — Google Calendar freebusy is the source of truth for double-booking prevention

---

## 3. Date/Timeframe Detection (minimal regex — let Claude handle the rest)

Only use regex to detect WHEN to re-fetch slots (what date range the user wants). Never use regex to parse which slot they're choosing — Claude handles that.

Timeframe signals to detect in `lastUserMsg`:
- `"next week"` → `searchFromDate` = today + 7 days, `daysWindow = 7`
- `"next month"` → `searchFromDate` = 1st of next month, `daysWindow = 14`
- `"couple/few weeks"` → `searchFromDate` = today + 14 days, `daysWindow = 7`
- `"in N weeks/months"` → calculate accordingly
- Specific date (`"March 10"`, `"March 10th"`, `"the 10th"`, `"10th"`) → `searchFromDate` = that exact date, `daysWindow = 1`
- Vague month (`"sometime in April"`) → `searchFromDate` = 1st of that month, `daysWindow = 14`
- Confirmation messages (`"yes"`, `"that works"`, `"sounds good"`) with NO date → no re-fetch, use cache

**Important:** Bare numbers like "at 2" or "10am" must NOT be treated as dates. Only trigger date detection when a month name OR ordinal suffix (`st/nd/rd/th`) is present.

## 6. BOOK Command Parser

When ARIA outputs `BOOK:{...}`:

1. Parse the JSON
2. Look up slot from `conversationSlots.get(convId)?.slots`:
   - **PRIMARY**: exact `slotLabel` match
   - **FUZZY**: strip EDT/EST suffix and match
   - **FUZZY**: match on date part + time part separately
   - **FALLBACK**: use `slotIndex` (1-based → 0-based)
   - **LAST RESORT**: `slots[0]`
3. Log: `📌 Booking slot: [label] | UTC: [start] | ARIA said: [slotLabel]`
4. Call `bookAppointment()`

## 7. `bookAppointment()` — Google Calendar Event

```js
{
  summary: `Consultation: ${name} (${company}) x NeuralFlowAI`,
  description: `Company: ${company}\nPain Points: ${notes}\nBooked via ARIA.`,
  start: { dateTime: slotStart, timeZone: 'America/New_York' },
  end:   { dateTime: slotEnd,   timeZone: 'America/New_York' },
  attendees: [{ email: GMAIL_USER }],  // Danny only — no client in attendees
  sendUpdates: 'none',                  // We send our own emails via Resend
  conferenceData: { createRequest: { requestId: `nf-${Date.now()}`, conferenceSolutionKey: { type: 'hangoutsMeet' } } },
  conferenceDataVersion: 1,
}
```

**Why no client in attendees:** We send a custom-branded email to the client via Resend instead. The Google Calendar invite contains pain points/notes which should NOT be visible to the client.

Retry logic: 3 attempts, exponential backoff (2s, 4s, 8s).
